# Wgame 周会数据报告生成技能

## 触发条件

当用户提到以下任何关键词时触发:
- "wgame周会"
- "战火勋章周报"
- "wgame周报"
- "周会数据"
- "weekly report"
- "Wgame 周报"
- "战火勋章周会"

## 定时任务设置

**自动执行时间**:每周二上午10:00

**产出内容**:
1. 从DAP报表#16621拉取数据
2. 生成Markdown格式报告
3. 上传到飞书Wiki指定位置:https://lilithgames.feishu.cn/wiki/ViqBw7xqRiXd3pkC5jdcXo6nnof
4. 发送简报给用户私聊

## 执行流程

### 步骤 1: 计算日期范围

自动计算以下日期范围(UTC+0 时区):
- **上周**:完整的周一 00:00:00 到周日 23:59:59
- **上上周**:完整的周一 00:00:00 到周日 23:59:59
- **上上上周**:完整的周一 00:00:00 到周日 23:59:59

**日期计算规则**(关键:产出时间是周二,数据周期是完整的周一到周日):
- 上周 = 本周二往前推8天(上周一)到本周二往前推2天(上周日)
- 上上周 = 上周一往前推7天(上上周一)到上周一往前推1天(上上周日)
- 上上上周 = 上上周一往前推7天(上上上周一)到上上周一往前推1天(上上上周日)

**⚠️ 重要:日期验证要求**
- 任何日期计算必须通过代码验证,禁止凭直觉回答
- 示例日期必须与实际日历核对
- 不确定时先验证再回答

使用 Python 脚本计算日期:
```python
from datetime import datetime, timedelta

today = datetime.utcnow()
# 今天是周二,往前推8天到上周一
today_weekday = today.weekday()  # 周二 = 1

# 上周(完整的周一到周日)
last_monday = today - timedelta(days=today_weekday + 7)  # 往前推8天
last_sunday = last_monday + timedelta(days=6)

# 上上周(完整的周一到周日)
week_before_monday = last_monday - timedelta(days=7)
week_before_sunday = week_before_monday + timedelta(days=6)

# 上上上周(完整的周一到周日)
two_weeks_before_monday = week_before_monday - timedelta(days=7)
two_weeks_before_sunday = week_before_monday + timedelta(days=6)
```

**示例**(假设今天是 2026-05-12 周二):
- 上周:2026-05-05 ~ 2026-05-11(上周一到上周日)
- 上上周:2026-04-27 ~ 2026-05-03(上上周一到上上周日)
- 上上上周:2026-04-20 ~ 2026-04-26(上上上周一到上上上周日)

### 步骤 2: 数据查询

**必须使用 DAP 自定义报表 #16621**(Wgame-全渠道-月度-lin)

**查询参数规范**:
- report_id: 16621
- tz: 0 (UTC+0)
- 使用 atlas-ai-gateway 服务的 DAP 工具

**必须查询的表**:

| 数据类型 | 表名 | 平台筛选 | 时间范围 |
|---------|------|---------|---------|
| 上上周汇总 | store | 无 | 上上周(week_before_monday ~ week_before_sunday) |
| 上上上周汇总 | store | 无 | 上上上周(two_weeks_before_monday ~ two_weeks_before_sunday) |
| 上周汇总 | store | 无 | 上周(last_monday ~ last_sunday) |
| 分平台数据 | store | 无 | 上周 |
| Android分国家 | country | filter_platform=android | 上周 |
| iOS分国家 | country | filter_platform=ios | 上周 |
| Android分渠道 | media_src | filter_platform=android | 上周 |

**重要**:汇总数据必须使用 `store` 表查询,然后将 Android + iOS + PC 三平台数据相加得到总计。禁止使用 `day` 表查询汇总数据(会导致数据翻倍或计算错误)!

**关键字段说明**:
- `总新增账号数`:DAP返回的字段,不是"安装数"
- `cost_per_install`:DAP返回的"新增成本"字段,直接使用,不要手动计算
- `roi_1`:首日ROI
- `roi_7`:7日ROI
- `actual_roi`:实际ROI(上周数据)

**ROI_7/ROI_1计算规则**:
- 必须使用store表**总计行**(不分平台)的ROI_7和ROI_1计算
- 计算公式:ROI_7 / ROI_1
- **禁止**使用分平台数据计算后再平均

**重要**:
1. 分国家数据和分渠道数据只查询上周(前一周),不要查询两周之和!
2. DAP 报表默认返回20条数据,使用 `page_size=300` 参数可以获取更多数据(最多300条)
3. 查询 country 表时必须添加 `page_size=300` 参数,确保获取完整的 Top10 数据
4. **查询上上周数据时也必须使用 `page_size=300`**,确保获取所有国家数据用于计算环比

### 步骤 3: 数据处理

**计算指标**:
- 总新增成本 = 总消耗 / 总新增
- 环比变化百分比 = (本期 - 上期) / |上期| * 100%
- 变化绝对值 = 本期 - 上期

**排序规则(重要)**:
- **消耗 Top10 国家**:必须按消耗降序排列(从高到低)
- **新增 Top10 国家**:必须按新增降序排列(从高到低)

**环比数据匹配规则(关键)**:
1. 查询上周Top10国家列表
2. 在上上周完整数据中查找这些国家的对应数据
3. **必须确保每个国家都找到上上周数据**,如果找不到,说明page_size不够大或查询有误
4. 对于新进入Top10的国家(上上周不在Top10),需要单独查询其上上周数据
5. **禁止**将找不到数据的标注为"-",必须确认数据确实存在

**识别需要标记的数据**:
- 环比变化超过 ±20% 的指标:加粗显示
- **Actual_ROI 最低的三个国家**:从「消耗 Top10」和「新增 Top10」的并集中筛选 Actual_ROI 最低的三个国家(排除 null 值),加粗显示

**KPI 指标(必须在对应表格中添加)**:
| 数据类型 | 指标 | KPI 绝对值 | 达标判断 |
|---------|------|-----------|---------|
| 上上周汇总 | 整体 ROI_7 | 6.8% | 实际值 ≥ 6.8% 为达标 |
| 上上周汇总 | ROI_7/ROI_1 | 4 | 实际值 ≥ 4 为达标 |
| 上周汇总 | 整体 Actual_ROI | 4.1% | 实际值 ≥ 4.1% 为达标 |
| 分平台数据 | Android Actual_ROI | 5.4% | 实际值 ≥ 5.4% 为达标 |
| 分平台数据 | iOS Actual_ROI | 3.6% | 实际值 ≥ 3.6% 为达标 |
| 安卓分渠道 | Google Actual_ROI | 3.4% | 实际值 ≥ 3.4% 为达标 |
| 安卓分渠道 | Facebook Actual_ROI | 2.8% | 实际值 ≥ 2.8% 为达标 |
| 安卓分渠道 | Tiktok Actual_ROI | 4.0% | 实际值 ≥ 4.0% 为达标 |
| 安卓分渠道 | Almedia Actual_ROI | 8.0% | 实际值 ≥ 8.0% 为达标 |

**KPI 展示规则**:
- 在对应表格中增加「KPI」和「是否达标」两列
- 未达标的指标在表格下方标注出来,格式:"⚠️ **XXX 未达标**:实际值 X.XX%,KPI 要求 X.X%"

**KPI 更新提醒**:
- 每月10号早上10点自动询问用户最新KPI数值
- 如果用户未同步新KPI,则在10号后的每天早上10点继续询问,直到获得更新
- KPI数值更新后,同步更新到本SKILL.md文档的KPI指标表中

## 关键规则（务必遵守）

1. **数据源**：必须使用 DAP 自定义报表 #16621
2. **时区**：所有数据使用 UTC+0 时区
3. **平台筛选（关键）**：
   - 分国家数据必须分别查询 Android 和 iOS（使用 filter_platform 参数）
   - 分渠道数据必须分别查询 Android 和 iOS
   - **Android和iOS的数据完全独立，禁止混用**
4. **时间范围**：
   - 分国家数据只查询上周（前一周），不是前两周之和
   - 分渠道数据只查询上周（前一周），不是前两周之和
5. **排除指标**：不要包含总充值流水和DAU
6. **报告结构**：不要包含总结与建议部分
7. **数据核对（关键）**：
   - 生成报告前，必须核对关键指标（Actual_ROI、ROI_1等）是否与查询结果一致
   - 对于跨平台的国家（如US、GB），必须确认使用的是正确平台的数据

## 常见错误与避免方法

### ❌ 错误1：平台数据混淆（最严重）
**原因**：将Android的数据错误地用于iOS，或反之
**典型案例**：
- Android US Actual_ROI = 6.00% → 错误地用于iOS US
- Android GB Actual_ROI = 3.52% → 错误地用于iOS GB
**正确做法**：
- **Android和iOS的数据必须完全独立**
- 即使国家代码相同（如US），两个平台的Actual_ROI也是不同的数据
- 查询时必须明确指定filter_platform参数
- 生成报告时，必须严格区分两个平台的数据，禁止混用

### ❌ 错误2：关键指标未核对
**原因**：生成报告前没有核对关键指标（如Actual_ROI）是否与查询结果一致
**正确做法**：
- 生成报告前，必须将报告中的Actual_ROI与原始查询结果进行交叉验证
- 对于异常值（如iOS FR Actual_ROI 17.99%），必须确认数据来源
- 保存每次查询的原始结果，便于追溯

### ❌ 错误3：ROI_7/ROI_1计算错误
**原因**：使用了分平台的ROI_7/ROI_1，而不是整体值
**正确做法**：
- 使用store表**总计行**（不分平台）的ROI_7和ROI_1计算
- 计算公式：ROI_7 / ROI_1
- **禁止**使用分平台数据计算后再平均

### ❌ 错误4：上上周数据遗漏
**原因**：page_size不够大，导致上上周数据查询不完整
**正确做法**：
- 查询上上周数据时必须使用 `page_size=300`
- 仔细核对上周Top10国家是否在上上周数据中都能找到
- 对于新进入Top10的国家，需要确认其上上周数据是否存在

### ❌ 错误5：数据字段混淆
**原因**：将"安装数"与"总新增账号数"混淆
**正确做法**：
- 使用DAP返回的"总新增账号数"字段
- 使用DAP返回的"cost_per_install"（新增成本）字段
- **禁止**手动计算新增成本

### ❌ 错误6：环比数据匹配错误
**原因**：没有正确匹配上周和上上周的国家数据
**正确做法**：
1. 查询上周Top10国家列表
2. 在上上周完整数据中查找这些国家的对应数据
3. **必须确保每个国家都找到上上周数据**
4. 对于新进入Top10的国家，需要单独查询其上上周数据

## 错误处理

如果遇到以下情况,向用户说明:
1. DAP 数据查询失败 - 检查网络连接和权限
2. 飞书文档创建失败 - 检查 Wiki 节点权限
3. 数据缺失 - 标记为 "-" 并在报告中说明
